  <name>Task 3: Simular validación de notarización (spctl) para verificar que no habrá rechazos preliminares</name>
  <files>ExtractorApp/ExtractorApp/build/Release/ExtractorApp.app</files>
  <action>
    El comando spctl simula los controles que Apple hace durante la notarización. Aunque una notarización real requiere enviar a Apple, podemos verificar que el binario cumple los requisitos básicos:

      spctl -a -v /Volumes/ESSAGER/__01.-Proyectos/__Herramientas_Desktop/extractor-url/ExtractorApp/ExtractorApp/build/Release/ExtractorApp.app

    Esperado: Output que indica "accepted" o "will be allowed" bajo las políticas de seguridad de macOS. Si la app no está notarizada (que es nuestro caso para desarrollo local), spctl puede devolver "rejected" o un mensaje similar, pero esto es normal para un binario sin notarización. Lo importante es que no haya un error de configuración de entitlements que haría que spctl lo rechace categóricamente.

    En desarrollo local (sin notarización Apple), el usuario puede ejecutar:
      sudo spctl --master-disable
    para permitir apps no notarizadas temporalmente, o confiar en la app en System Preferences.

    Captura el output de spctl. Un resultado es aceptable siempre que no esté relacionado con entitlements malformados.
  </action>

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-07-05 | Tampering | Entitlements + Firma | mitigate | codesign valida que entitlements son coherentes entre .entitlements y binario |
| T-07-06 | Integrity | Configuración Xcode | mitigate | Grep cuenta correctas ocurrencias de MACOSX_DEPLOYMENT_TARGET y ARCHS |
